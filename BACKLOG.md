# Backlog

Этот файл — парковка идей, UX-замечаний, багов ручной проверки и технического долга, которые не входят в текущий PR.

Правила:

- Кодовое слово `В бэклог:` означает: зафиксировать пункт здесь, но не реализовывать его без отдельного запроса.
- Кодовое слово `Из бэклога:` означает: найти пункт здесь и подготовить его к обсуждению или отдельной реализации.
- Запись в backlog не меняет текущий этап проекта.
- `PROJECT_STATE.md` остается единым источником истины о текущем статусе проекта.

Статусы backlog:

- `Зафиксировано` — пункт записан, но не взят в работу.
- `Обсуждается` — пункт требует уточнения решения.
- `Запланировано` — пункт выбран для будущего PR.
- `Закрыто` — пункт реализован, отменен или стал неактуален.

## BL-001 — Кнопка “Повторить обработку” видна во время активного созвона

- Тип: UX / safety bug
- Статус: Закрыто
- Приоритет: средний
- Где найдено: `Рабочий день` → `Встречи за день` → раскрытая карточка созвона

### Описание

Если созвон идет и запись активна, внутри карточки встречи уже видна кнопка `Повторить обработку`. Это может быть опасно или нелогично: пользователь может нажать ее до завершения записи или pipeline.

### Ожидаемое поведение

- Не показывать кнопку во время активного созвона.
- Показывать ее только если обработка завершилась с ошибкой.
- Либо показывать после полного завершения обработки, если повторная обработка действительно безопасна.

### Решение

Закрыто в ветке `codex/safe-reprocess-button`: кнопка `Повторить обработку` больше не создается в раскрытой карточке встречи для активной встречи, встречи в очереди, встречи в обработке, встречи без записи и встречи со статусом `Итоги выключены`. Кнопка показывается только для статусов `Требует внимания` и `Итоги готовы`, если есть доступный файл записи, с которого можно безопасно повторить pipeline.

## BL-002 — Проверить корректность блока “Готовность системы” и авто-проверки

- Тип: UX / reliability
- Статус: Закрыто
- Приоритет: средний
- Где найдено: `Рабочий день` → `Готовность системы`

### Описание

Нужно проверить, что блок `Готовность системы` действительно отображает все важные проверки и правильные статусы.

### Что рассмотреть

- Корректность статусов OBS, FFmpeg, backend транскрипции, Summary, API key и endpoint.
- Автоматический запуск проверки готовности при старте приложения.
- Автоматический запуск проверки готовности при начале рабочего дня.
- Возможный запуск перед первой встречей за день.
- Не запускать проверку перед каждой встречей, чтобы не замедлять сценарий.

### Частичное решение

Ветка `codex/async-readiness-checks`: поведение кнопки `Проверить готовность` переведено на отдельный worker/thread.

Реализовано:

- `Проверить готовность` больше не блокирует UI на OBS/WebSocket и других медленных проверках.
- На время проверки показывается состояние `Проверяется...`, кнопка временно отключается, карточки обновляются после завершения.
- Если пользователь сохраняет настройки во время проверки, старый результат не перезаписывает карточки: UI просит запустить проверку повторно.
- После `Сохранить настройки` дополнительно показывается, что текущая проверка выполнялась со старыми настройками.
- Закрытие приложения во время проверки готовности временно блокируется понятным статусом, чтобы не уничтожать активный Qt worker/thread.
- Состав readiness-карточек не менялся после PR #50.

### Решение

Закрыто в ветке `codex/readiness-autocheck-rules`: проверка готовности автоматически запускается при показе главного окна, после сохранения настроек и после начала рабочего дня. Если пользователь меняет настройки во время текущей проверки, старый результат помечается устаревшим, а повторная проверка запускается автоматически после завершения текущей.

Перед стартом встречи используется последняя актуальная проверка готовности. Если проверка еще не выполнялась или устарела, приложение запускает проверку и просит дождаться результата. Ошибка `Запись разговора (OBS)` блокирует старт встречи, потому что без OBS основной сценарий приложения теряет смысл. Ошибки `Извлечение аудио (FFmpeg)` и `Транскрипция` не блокируют запись, но показывают предупреждение: пользователь может осознанно начать встречу, понимая, что дальнейшая обработка может потребовать ручного исправления или повторного запуска.

## BL-004 — Вынести переключение темы в левую панель

- Статус: Запланировано

- Где найдено: левая навигационная панель приложения

Floating control продолжает использовать уже существующую настройку `ui.floating_theme`: если стоит `Как в приложении`, он наследует новую тему.

## BL-005 — Доработать настройки хранения и выбор папки данных

- Тип: UX / setup / reliability
- Статус: Закрыто
- Приоритет: высокий
- Где найдено: `Настройки` → `Хранение` → `Папка данных`

### Описание

Сейчас папка данных задается обычным текстовым полем. Это неудобно и рискованно: пользователь может ошибиться в пути.

### Ожидаемое поведение

- При первой установке или первом запуске создается понятная папка данных по умолчанию.
- В настройках рядом с путем есть кнопка `Выбрать папку`.
- Открывается стандартное окно выбора папки Windows.
- После выбора путь сохраняется в `config.yaml`.
- Приложение проверяет, что папка существует и доступна для записи.
- Если папки нет, приложение может предложить создать ее.

### Открытые вопросы

- Какой путь считать дефолтным, например `D:\MeetingsApp\data\MeetingSummaries`.
- Нужна ли миграция старых данных при смене папки.
- Предупреждать ли пользователя, что старые встречи останутся в старом месте.
- Что делать, если выбранная папка недоступна.

### Решение

Закрыто в ветке `codex/storage-folder-picker`: рядом с полем `Папка данных` добавлена кнопка `Выбрать папку`, которая открывает стандартный системный выбор папки. При сохранении настроек приложение создает выбранную папку, проверяет, что путь не указывает на файл, и делает короткую проверку записи. Если рабочий день или встреча уже активны, новый путь сохраняется в `config.yaml`, но текущий `StorageService` продолжает работать со старой папкой до завершения рабочего дня, чтобы не переключить активные данные на лету.

## BL-006 — Негативные сценарии: краши приложения, pipeline и OBS

- Тип: reliability / product logic / safety
- Статус: Зафиксировано
- Приоритет: высокий
- Где найдено: весь lifecycle рабочего дня, встречи, OBS и pipeline

### Описание

Нужно отдельно разобрать, как приложение должно вести себя в аварийных сценариях.

### Сценарии

- Приложение закрыли или оно упало во время активного созвона.
- Приложение упало во время pipeline: FFmpeg / транскрипция / summary / итоги дня.
- OBS упал или отключился во время записи.
- OBS WebSocket стал недоступен.
- Запись не стартовала, но встреча считается активной.
- Запись стартовала, но путь к файлу не вернулся.
- Запись есть, но аудио не извлеклось.
- Транскрипция или summary упали после частичного выполнения.

### Отдельный вопрос

Решение по галке `OBS включен`: пользовательскую возможность отключить OBS нужно убрать. Если OBS выключен, приложение теряет основную ценность: нет записи, аудио, транскрипции и summary. Для обычного пользователя OBS должен быть обязательной частью готовности системы.

Ветка `codex/remove-obs-disable-option`: пользовательская галка отключения OBS удаляется из `Настроек`, `config.yaml.example`, README и сохраняемого `config.yaml`. Внутренний `NoopRecorder` остается только тестовой подменой.

### Что решить

- Какие состояния восстанавливать после перезапуска.
- Как показывать “незавершенную встречу” после краша.
- Пытаться ли остановить OBS после восстановления.
- Как отличать падение приложения от обычного закрытия окна.
- Какие pipeline-шаги можно безопасно перезапускать.
- Когда показывать кнопку `Повторить обработку`.
- Нужно ли блокировать старт встречи, если OBS не готов.

## BL-007 — Визуализация ограничений и предупреждения о рискованных действиях

- Тип: UX / safety / reliability
- Статус: Зафиксировано
- Приоритет: высокий
- Где найдено: основной UI, особенно `Рабочий день`, pipeline, закрытие приложения

### Описание

Часть ограничений в приложении может быть непонятна пользователю. Нужно явно объяснять, почему действие недоступно или рискованно.

### Примеры

- Почему нельзя нажать `Завершить рабочий день`, если есть активные pipeline.
- Почему нельзя завершить рабочий день при активной встрече.
- Почему нельзя начать встречу, если рабочий день не активен.
- Почему какие-то кнопки disabled.
- Что происходит, если закрыть приложение во время активного созвона.
- Что происходит, если закрыть приложение во время фоновой обработки.

### Ожидаемое поведение

- Рядом с disabled-кнопками или внутри блока показывать короткое объяснение.
- При наведении или клике на недоступное действие можно показывать подсказку.
- При закрытии приложения во время активного созвона показывать предупреждение.
- При закрытии приложения во время pipeline показывать предупреждение о возможных последствиях.
- Если все спокойно, закрывать приложение без лишнего шума.

### Открытые вопросы

- Какие действия блокировать жестко.
- Какие действия разрешать после подтверждения.
- Нужна ли кнопка `Все равно закрыть`.
- Как восстанавливать состояние после закрытия во время pipeline.
- Где показывать причины блокировки: карточка, tooltip, status bar или modal.

### Частичное решение

Ветка `codex/visible-safety-prompts`: для первого PR по BL-007 согласованы и реализуются предупреждения при закрытии приложения и видимые причины ограничений в блоках `Состояние дня` и `Активный созвон`. Активный созвон блокирует закрытие приложения и показывает модальное предупреждение; фоновая обработка показывает выбор `Остаться в приложении` / `Закрыть и восстановить позже`. Завершение рабочего дня при фоновой обработке остается разрешенным, но UI явно объясняет, что `Итоги дня` начнутся после завершения обработки встреч.

## BL-009 — Непонятный статус pipeline между этапами обработки

- Тип: UX / pipeline clarity
- Статус: Закрыто
- Приоритет: высокий
- Где найдено: карточка встречи → `Pipeline этой встречи`

### Описание

Возможна ситуация: `Аудио` уже готово, но `Транскрипция` показывает `Ожидает` и текст `Ждет audio.wav`. Визуально это нелогично: аудио уже извлечено, но следующий этап будто ждет файл.

### Ожидаемое поведение

- Если `audio.wav` уже готов, но транскрипция еще не началась, показывать `Ожидает очереди` или `Ждет запуска транскрипции`.
- Если сейчас обрабатывается другая встреча, писать `Ожидает очереди: сейчас обрабатывается другая встреча`.
- Если backend готовится, загружает модель или отправляет chunk, показывать отдельный активный статус.
- Не писать `Ждет audio.wav`, когда `audio.wav` уже создан.

### Что проверить

- Как формируется pipeline-статус из metadata.
- Различаются ли состояния `не хватает файла`, `ждет очереди`, `выполняется`, `ошибка`.
- Нужно ли добавить отдельные metadata-поля для очереди.
- Как это будет выглядеть для AI Tunnel chunking.

### Решение

Закрыто в ветке `codex/pipeline-status-model`: pipeline больше не показывает `Ждет audio.wav`, если аудио уже извлечено. Для завершенных встреч с `processing_status = pending` используется понятный статус `Ждет обработки встречи`; сегодняшние pending-встречи восстанавливаются в очередь при запуске приложения.

## BL-010 — Статус встречи должен показывать “Требует внимания”, если pipeline завершился с ошибкой

- Тип: UX / status model / reliability
- Статус: Закрыто
- Приоритет: высокий
- Где найдено: `Рабочий день` → `Встречи за день`, `Ревью` → карточки встреч

### Описание

Сейчас если сама встреча завершена, карточка может показывать статус `Завершена`, даже если внутри pipeline есть ошибка. Пример: встреча завершилась как запись, но транскрипция упала, а summary пропущен.

### Ожидаемое поведение

- Если `status = ended`, но важный pipeline-этап имеет ошибку, карточка должна показывать `Требует внимания`.
- В раскрытой карточке pipeline должен показывать конкретный проблемный этап.
- `summary_status = disabled` не должен считаться ошибкой, если summary выключен намеренно.
- `transcription_quality = suspect` возможно тоже должен переводить встречу в `Требует внимания`.

### Что обсудить

- Какие статусы считать критичными.
- Какие пропуски допустимы.
- Нужно ли сортировать встречи `Требует внимания` выше остальных.
- Показывать ли кнопку `Повторить обработку` только для таких встреч.

### Решение

Закрыто в ветке `codex/pipeline-status-model`: карточки встреч используют пользовательские статусы результата (`Итоги готовы`, `Итоги выключены`, `Без записи`, `Ждет обработки встречи`, `Обработка`, `Требует внимания`) вместо общего финального статуса `Завершена`. Ошибки pipeline, `summary_status = skipped` и `transcription_quality = suspect` переводят встречу в `Требует внимания`; намеренно выключенный summary показывает `Итоги выключены`.

## BL-011 — Заменить кнопки папок в карточке встречи на “Открыть итоги встречи”

- Тип: UX / navigation
- Статус: Закрыто
- Приоритет: средний

### Ожидаемое поведение

- Вместо `Открыть папку встречи` и `Открыть папку дня` показать одну кнопку `Открыть итоги встречи`.
- Кнопка показывается только если итог встречи реально готов: `summary_status = draft_created`, `summary_draft.md` существует и не является placeholder.
- По нажатию приложение переключается в раздел `Ревью`.
- Выбирается конкретный созвон.
- Открывается вкладка `Итоги встречи`.

### Решение

Закрыто в ветке `codex/open-meeting-summary-action`: файловые кнопки `Открыть папку встречи` и `Открыть папку дня` убраны из раскрытой карточки встречи на вкладке `Рабочий день`. Вместо них показывается `Открыть итоги встречи`, но только для встреч с настоящим готовым summary. Если summary отсутствует, является placeholder, еще обрабатывается, выключен настройками или pipeline требует внимания, кнопка перехода к итогам не показывается. При нажатии приложение открывает `Ревью`, выбирает конкретную встречу и ставит активной вкладку `Итоги встречи`.

## BL-012 — Реализовать раздел “Архив” для просмотра прошлых дней внутри приложения

- Тип: feature / UX / navigation
- Статус: Зафиксировано
- Приоритет: высокий
- Где найдено: раздел `Архив`

### Описание

Сейчас `Архив` — заглушка. Нужно сделать полноценный раздел, чтобы пользователь мог смотреть прошлые встречи и итоги дня без ручного открытия папок на ПК.

### Ожидаемое поведение

- В разделе `Архив` есть выбор дня через календарь.
- В календаре заранее видно, где есть рабочий день, встречи или данные отсутствуют.
- После выбора дня открывается UI, похожий на `Ревью`.
- Явно показано, какой день сейчас просматривается.
- Можно открыть список встреч выбранного дня.
- Можно открыть итоги конкретной встречи.
- Можно открыть transcript конкретной встречи.
- Можно открыть `Итоги дня`, если они созданы.

- Как визуально обозначать дни в календаре: есть встречи, день завершен, есть ошибки pipeline, есть итоги дня.

- Нужна ли фильтрация по статусам: `готово`, `требует внимания`, `без итогов`.
- Разрешать ли повторную обработку прошлых встреч из архива или только просмотр.
- Должен ли архив быть read-only или с действиями.
- Как показывать день, который был переоткрыт несколько раз.

## BL-016 — Корректная работа фильтров Архива по периоду и диапазону дат

- Тип: UX / archive filters / reliability
- Статус: Закрыто
- Приоритет: высокий
- Где найдено: `Архив` → быстрые фильтры `Неделя` / `Месяц` / `Все` и поля дат `с` / `по`

### Описание

Фильтры Архива должны работать только по прошлым рабочим дням и не показывать сегодняшний день. Быстрые периоды должны иметь календарные границы до вчера включительно, а ручное редактирование дат должно быть явно отделено от быстрых фильтров.

### Ожидаемое поведение

- `Неделя` показывает последние 7 календарных дней до вчера включительно.
- `Месяц` показывает текущий календарный месяц с 1-го числа по вчера включительно.
- `Все` показывает все прошлые рабочие дни, верхняя граница по умолчанию — вчера.
- Поле `по` по умолчанию заполнено вчерашней датой; сегодняшняя или будущая дата нормализуется до вчера.
- При ручном редактировании `с` или `по` быстрые фильтры скрываются, поля дат подсвечиваются, появляется статус `Ручной диапазон` и кнопка `Сбросить`.
- `Сбросить` возвращает `Все`, очищает `с` и ставит `по` равным вчера.
- Поиск работает только внутри выбранного периода или ручного диапазона.
- Пустые результаты и невалидный диапазон `с > по` показывают понятное состояние без падения интерфейса.

### Текущее решение

PR #69, ветка `codex/archive-date-filters`: подготовлена реализация через TDD. Быстрые периоды Архива получают календарные границы до вчера включительно, стандартное состояние `Все` заполняет `по` вчерашней датой, ручной диапазон отделен от быстрых фильтров визуальным состоянием и кнопкой `Сбросить`, а поиск остается внутри выбранного диапазона. Отдельная кнопка `Диапазон` не добавлялась.
