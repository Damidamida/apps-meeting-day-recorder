# Meeting Day Recorder — состояние проекта

## 1. Цель продукта

Локальное Windows-приложение для ручной фиксации ad hoc-созвонов в течение рабочего дня, последующей транскрипции, подготовки итогов по каждой встрече и общих итогов дня, ревью пользователем и сохранения финальных файлов локально.

## 2. Основной сценарий

Пользователь начинает рабочий день, вручную запускает и завершает встречи, которые часто появляются на лету из чатов, завершает рабочий день, проверяет черновики итогов, редактирует их и сохраняет финальные файлы.

## 3. Важные ограничения

- В MVP нет интеграции с почтой, календарем и мессенджерами.
- Встречи часто формируются ad hoc из чатов.
- Основной сценарий MVP — ручные кнопки `Start workday`, `Start meeting`, `End meeting`, `End workday`.
- Хранение данных остается local-first.
- Нельзя коммитить записи, аудио, транскрипты с персональными данными, `config.yaml`, `.env` и секреты.
- OBS, ffmpeg, OpenAI API и AI-суммаризация добавляются только на соответствующих этапах.
- Нельзя пушить напрямую в `main`.
- Работа идет через отдельные ветки `codex/*` и PR.
- Весь пользовательский текст должен быть на русском языке.

## 4. Правила языка

- Заголовки и описания PR — на русском.
- `README.md` и `PROJECT_STATE.md` — на русском.
- Пользовательский UI — на русском.
- Ошибки и статусы приложения — на русском.
- Метки экрана ревью, шаблоны итогов, метки задач и заголовки генерируемых Markdown-файлов — на русском.
- Английский допустим только для кода, технических команд, библиотек, внешних API и обязательных технических литералов.

## 5. Правила работы Codex с контекстом

- Перед задачей Codex обязан сверить репозиторий, базовую ветку, текущий этап и ограничения.
- После завершенной задачи Codex обязан обновить `PROJECT_STATE.md` в той же ветке и том же PR.

PR #34, ветка `codex/day-summary-pipeline`, был смержен после ручной проверки пользователем: `Итоги дня` стали отдельной карточкой в блоке `Встречи за день` на вкладке `Рабочий день` и первой карточкой в списке на вкладке `Ревью`. Карточка `Итоги дня` получила собственный pipeline: сбор итогов встреч, проверка наличия summary, генерация выжимки через OpenAI-compatible endpoint / ProxyAPI и формирование ссылок на transcript встреч внутри приложения.

Автоматическое формирование `Итогов дня` запускается после нажатия `Завершить рабочий день`. Если обработка встреч еще идет в фоне, `Итоги дня` переходят в состояние ожидания и стартуют после завершения очереди. Кнопка `Обновить итоги дня` доступна только в карточке `Итоги дня` на вкладке `Рабочий день`. При повторном завершении переоткрытого дня итог не пересобирается без новых встреч; если появились новые встречи, существующий черновик дополняется через OpenAI-compatible endpoint / ProxyAPI.

Для дневных итогов во внешний сервис отправляются только тексты `summary_final.md` или `summary_draft.md` встреч. Аудио, видео и transcript во внешний сервис для дневных итогов не отправляются. Если у встречи отсутствует summary, она все равно включается в дневной итог с явной пометкой об отсутствии summary.

Договоренность по задачам дня сохраняется: функционал задач дня скрыт из UI, но существующая файловая логика `00_tasks_draft.md` / `00_tasks_final.md` не удаляется, чтобы не ломать старые данные.

## 13. Архитектурные решения

- Python 3.11+.
- PySide6 для desktop UI.
- YAML для конфигурации.
- JSON для метаданных.
- Markdown для итогов.
- OBS WebSocket для локального управления записью встреч.
- FFmpeg для локального извлечения `audio.wav` из OBS-записи.
- Транскрипция выполняется локально через optional Whisper CLI или optional faster-whisper backend; optional AI Tunnel STT backend можно включить явно через `transcription.backend: aitunnel`. Для AI Tunnel длинный WAV-файл может автоматически нарезаться на части с отображением прогресса в pipeline.
- AI Tunnel / OpenAI-compatible endpoint используется для генерации `summary_draft.md` одной встречи из готового текстового transcript и для генерации `00_day_summary_draft.md` из текстов summary встреч, если summary явно включен в локальном `config.yaml`. ProxyAPI остается возможным альтернативным custom endpoint.
- Сначала ручной сценарий, автоматизация позже.

## 14. Известные риски и решения на пересмотр

- Восстановление состояния после перезапуска должно сохраняться.
- Выходные файлы должны оставаться вне git.
- Нужно принять решение о промежуточной папке `meetings/`.
- Настройка OBS-сцены и аудио будет ручной предпосылкой.
- Пароль OBS WebSocket хранится только в локальном `config.yaml`, который не добавляется в git.
- `config.yaml.example` остается безопасным шаблоном без пароля; локальный рабочий `config.yaml` создается рядом с ним и не заменяет шаблон в репозитории.
- Путь записи настраивается в OBS; приложение сохраняет `recording_path`, только если OBS возвращает его после остановки записи.
- Большие записи могут потребовать политику очистки.
- Длинное WAV-аудио для внешнего AI Tunnel STT backend нарезается на части перед отправкой. Если отдельная часть все равно превышает лимит 25 МБ или формат не поддерживает локальную нарезку, встреча должна получить понятный статус ошибки без потери локальных файлов.
- Локальный Whisper CLI на модели `base` может обрабатывать запись примерно в режиме real-time; optional faster-whisper добавлен как первый локальный путь ускорения. Более тонкая настройка моделей и будущие внешние STT-провайдеры остаются возможными будущими улучшениями.
- Качество prompt для итогов нужно будет улучшать итерационно.
- AI Tunnel / OpenAI-compatible API key должен храниться только во внешнем окружении или локальном `.env.local`, который не добавляется в git.
- Генерация итогов не должна отправлять во внешний AI endpoint аудио или видео; для итогов встречи отправляется только текстовый transcript, для итогов дня — только тексты summary встреч.
- Нельзя смешивать этапы и добавлять AI или OBS раньше плана.
- Пользовательские тексты UI и генерируемых Markdown-заглушек переведены на русский язык на этапе Review UI.
