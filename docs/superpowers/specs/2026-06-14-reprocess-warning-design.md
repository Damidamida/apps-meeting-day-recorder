# BL-013: предупреждения перед перезаписью ручных итогов

## Контекст

Повторная обработка встречи и принудительное обновление итогов дня могут заменить уже отредактированные пользователем черновики. Приложение не определяет наличие ручных правок и должно предупреждать всегда.

## Решение

Перед запуском опасного действия показывается модальное подтверждение с кнопкой `Отмена` по умолчанию. Если пользователь отменяет действие, metadata, очередь и pipeline не меняются.

Для встречи перед действием `Повторить обработку`:

- заголовок: `Повторить обработку встречи?`;
- текст: `Если вы вручную меняли Итог встречи, новая обработка заменит ваши изменения.`;
- кнопки: `Повторить обработку` и `Отмена`;
- кнопка по умолчанию: `Отмена`.

Для итогов дня перед действием `Обновить итоги дня`:

- заголовок: `Обновить итоги дня?`;
- текст: `Если вы вручную меняли Итог дня, обновление заменит ваши изменения.`;
- кнопки: `Обновить итоги дня` и `Отмена`;
- кнопка по умолчанию: `Отмена`.

## Архитектурный подход

В UI используется небольшой общий in-app overlay/helper подтверждения опасного действия. Он принимает заголовок, текст и подпись подтверждающей кнопки, чтобы будущий `Архив` мог переиспользовать тот же механизм.

Для BL-013 не используется нативный системный `QMessageBox`: предупреждение должно выглядеть как внутренняя модалка приложения с затемнением фона, карточкой в стиле существующего UI и читаемым текстом в темной теме.

Утвержденные формулировки предупреждений должны быть основным видимым текстом модалки. Их нельзя заменять на более общие фразы, переносить в status bar, tooltip или metadata.

Существующие safety-проверки доступности действия выполняются до предупреждения. Предупреждение не должно открывать путь к недоступным сценариям.

## Вне scope

- Резервные копии.
- Восстановление предыдущей версии.
- Сравнение ручных правок.
- История версий.
- Изменение pipeline.
- Реализация `Архива`.
